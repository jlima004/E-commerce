## 6. Protocolo comum de preflight

Antes de cada plano:

1. executar `git fetch origin --prune`;
2. confirmar a branch exata;
3. capturar o HEAD documental atual;
4. confirmar paridade com a branch remota;
5. confirmar worktree limpo;
6. confirmar `git diff --check` vazio;
7. confirmar que o PLAN correspondente pertence ao HEAD;
8. confirmar que todos os SUMMARYs predecessores pertencem ao HEAD;
9. confirmar a decisão humana adequada do predecessor;
10. confirmar a allowlist de escrita do plano;
11. confirmar que PLAN/VALIDATION permanecem byte-identical;
12. parar antes da primeira operação externa e solicitar a autorização literal
    exigida pelo PLAN.

Comandos mínimos no fish:

```fish
rtk git fetch origin --prune
rtk git branch --show-current
rtk git rev-parse HEAD
rtk git status --short --untracked-files=all
rtk git diff --check
rtk git rev-list --left-right --count \
  origin/gsd/phase-12.1-mvp-release-readiness-production-validation...HEAD
```

Resultado requerido:

```text
branch: gsd/phase-12.1-mvp-release-readiness-production-validation
origin phase branch...HEAD: 0 0
worktree: empty
git diff --check: empty
```

Exceção única: durante o plano atual, o SUMMARY em construção pode deixar o
worktree temporariamente não limpo. O plano seguinte não começa antes de
revisão humana, commit autorizado, push humano/autorizado, paridade `0 0` e
worktree limpo.

## 7. Protocolo de SUMMARY

Cada plano materializa exatamente um SUMMARY. Ele deve conter:

```text
plan
execution baseline
branch
documentary HEAD
candidate SHA when applicable
authorizations received
tasks performed
commands or surfaces
evidence classes
sanitized results
negative proofs
limitations
stop conditions encountered
PASS or BLOCKED
files changed
sensitive values displayed
next permitted gate
```

Regras de persistência:

- persistir somente campos allowlisted pelo PLAN;
- nunca copiar saída bruta;
- nunca armazenar config-var values, URLs credenciadas, headers, tokens,
  cookies, payloads, PII, client secrets, Pix, PAN ou tracking token;
- classificar cada item com uma classe exata:
  `local process`, `local PostgreSQL disposable`, `Heroku release`,
  `production read-only`, `production mutation`, `real external provider` ou
  `inferred only`;
- distinguir execução atual, inferência e evidência histórica;
- registrar evidência histórica apenas como `historical context only`;
- listar explicitamente o único arquivo alterado e as negativas do plano.

Resultado literal:

```text
PLAN 12.1-0X: PASS
```

ou:

```text
PLAN 12.1-0X: BLOCKED
```

Em `BLOCKED`, produzir SUMMARY factual e parar. Não alterar runtime para fazer o
gate passar.

## 8. Protocolo de commit e push

Depois da parte técnica de cada plano:

1. parar;
2. mostrar SUMMARY, diff e verificações;
3. solicitar decisão humana;
4. não fazer commit antes da aprovação explícita;
5. após aprovação, adicionar somente o SUMMARY atual;
6. executar `git diff --cached --check` e mostrar o diff staged;
7. criar exatamente um commit contendo exatamente um SUMMARY;
8. não fazer push automaticamente;
9. parar e aguardar push humano ou autorização explícita;
10. após push, confirmar paridade `0 0` e worktree limpo;
11. somente então apresentar o plano seguinte para autorização.

Mensagens:

```text

docs(12.1-03): record authorized Heroku release
docs(12.1-04): record production read-only validation

docs(12.1-06): record final release readiness verification
```

Proibidos: `--amend`, rebase, autosquash, force push, multi-SUMMARY commit,
runtime commit e push automático.

## 13. Stop conditions

Parar o plano atual e impedir o seguinte quando houver:

- branch/paridade/worktree/predecessor/allowlist inválidos;
- PLAN/VALIDATION alterado;
- runtime/test/migration/package/config drift;
- candidate inexistente, não ancestral, mutado ou divergente do deployed SHA;
- Unit, Modules, HTTP, lint ou build falho;
- PostgreSQL descartável inseguro, remoto, compartilhado ou sem cleanup;
- inventário/schema manifest divergente;
- Gelato old/new identity mismatch;
- migration URL ausente/inválida/porta 6543;
- release contract ausente, divergente ou rollback target inelegível;
- Release Phase falho ou não DB-only;
- refspec rejeitada, release/SHA/formation inconsistente;
- health/dependency/SHA falho; rota essencial 5xx; Admin não fail-closed;
- raw log scan ausente/tardio, qualquer sensitive count maior que zero ou raw
  file remanescente;
- provider authorization/surface/sanitização ambígua;
- segredo, PII, payload ou URL credenciada na evidência;
- qualquer critério C01–C18 obrigatório falho.

Um finding bloqueia a fase somente quando há evidência de que ameaça payment,
Order ou fulfillment; expõe dados; quebra migration, build ou runtime; impede
deploy/health; viola invariável crítica; torna a validação insegura; invalida
candidate/release contract/rollback target; ou impede sanitização.

Não bloquear por preferência, refactor opcional, hardening não essencial,
melhoria estética, débito aceito ou provider opcional não exercitado.

## 14. Recovery policy

Se qualquer plano resultar em BLOCKED:

1. não iniciar o plano seguinte;
2. não alterar PLANs, runtime ou evidência anterior;
3. não “corrigir para passar” dentro deste fluxo;
4. não repetir operação externa;
5. não executar rollback;
6. não apagar evidência documental anterior;
7. produzir o SUMMARY factual do plano;
8. parar para revisão humana;
9. recomendar gate corretivo separado somente quando necessário.

Nova execução exige nova autorização humana explícita. Retry externo e rollback
automático são proibidos.

Recuperação local limita-se a limpar recursos temporários criados pelo plano.
Não usar reset, restore, stash, rebase ou amend. Um rollback real, se escolhido
em outro gate, exige autorização específica e nova validação de formação,
health, SHA, rotas e logs; nunca inclui `migration:down` automático.

## 15. Final verification protocol

Ao concluir tecnicamente cada plano:

```fish

rtk git diff --stat
rtk git diff --name-only
```

Exigir exatamente o SUMMARY atual como única mudança. Recalcular e comparar os
hashes dos seis PLANs e do VALIDATION com a seção 2.

Antes do 12.1-06 PASS, verificar:

- seis PLANs, seis waves, execução estritamente linear;
- seis SUMMARYs tracked, cada um em commit próprio e revisado;
- push não automático e paridade remota antes de cada sucessor;
- candidate do 12.1-01 propagado sem redefinição;
- deploy por refspec do candidate, nunca HEAD;
- `deployed_sha == candidate_sha`;
- `P12_RELEASE_CONTRACT_V1` consistente entre 02/03/06;
- rollback target igual à release atual pré-deploy, ainda elegível;
- Gelato hard stop preservado;
- raw scan fail-closed anterior à projeção e
  `sensitive_match_count = 0`;
- providers opcionais exercitados com autorização ou declarados como limitação;
- zero provider/business mutation fora do único deploy autorizado;
- zero runtime/test/migration/package/config change;
- C01–C18 e D12.1-01..D12.1-15 cobertos;
- STATE/ROADMAP inalterados durante 01–06.

Decisão final:

```text

```

somente com C01–C18 PASS; caso contrário:

