## Summary

O baseline já contém a arquitetura necessária para um release seguro: `Procfile` com processos `release`, `web` e `worker`; migration child DB-only; health separado em liveness/readiness; PostgreSQL e Redis no readiness; versão dinâmica compartilhada entre health e Sentry; e inicialização fail-fast dos quatro papéis Redis de produção. [VERIFIED: codebase inspection] A documentação oficial do Medusa confirma o build standalone em `.medusa/server`, o start a partir desse diretório, `db:migrate` para migrations de módulos e a separação de instâncias `server`/`worker`. [CITED: https://docs.medusajs.com/learn/deployment/general]

A Fase 12.1 não é uma nova rodada de implementação funcional. O plano futuro deve ser uma sequência de gates manuais que produzam evidência contemporânea: primeiro release candidate local, depois inventário/migration em PostgreSQL descartável, depois preflight e release Heroku, depois observação read-only e, por fim, decisão de PASS/BLOCKED com limitações explícitas. [VERIFIED: 12.1-CONTEXT.md] Nenhum resultado histórico de RC1, Fase 12 ou smoke de julho pode ser promovido a prova do release atual sem reexecução. [VERIFIED: 12.1-CONTEXT.md; inferred operational consequence]

O maior risco de planejamento é a migration Gelato. O baseline atual contém `Migration20260703000000.ts`, enquanto evidência histórica mostra que ela nasceu como `MigrationTBDGelatoFulfillment` e foi apenas renomeada, sem mudança factual de DDL. [VERIFIED: codebase and Phase 12 artifact inspection] O MikroORM 6.6.14 instalado registra a identidade da migration pelo nome do arquivo sem extensão; portanto, se o banco de produção tiver o nome antigo em `public.mikro_orm_migrations`, o nome atual será visto como outra migration pendente. [VERIFIED: installed @mikro-orm/migrations 6.6.14 implementation] A presença do nome antigo em produção não foi consultada e permanece desconhecida. [ASSUMED]

**Primary recommendation:** planejar seis gates manuais, com parada binária em qualquer incoerência de migration/release/health/topologia, e exigir uma inspeção read-only do histórico de migrations antes de autorizar o Release Phase. [VERIFIED: codebase inspection; CITED: https://devcenter.heroku.com/articles/release-phase]

### Recommended Project Structure

```text
.planning/phases/12.1-mvp-release-readiness-production-validation/
├── 12.1-CONTEXT.md             # decisões vinculantes existentes
├── 12.1-RESEARCH.md            # este artefato
├── 12.1-01-PLAN.md             # futuro: RC local e migration disposable
├── 12.1-02-PLAN.md             # futuro: preflight operacional
├── 12.1-03-PLAN.md             # futuro: release Heroku
├── 12.1-04-PLAN.md             # futuro: produção read-only
├── 12.1-05-PLAN.md             # futuro: observabilidade/limitações
└── 12.1-06-PLAN.md             # futuro: rollback/runbook e decisão final
```

A divisão acima é uma recomendação de planejamento; nenhum PLAN foi criado ou autorizado. [VERIFIED: current gate authorization]

### Pattern 1: Evidence-before-promotion

**What:** cada gate produz uma evidência contemporânea, classificada, sanitizada e suficiente antes de permitir o gate seguinte. [VERIFIED: 12.1-CONTEXT.md]

**When to use:** sempre; especialmente entre local, banco persistente, Heroku e providers. [VERIFIED: 12.1-CONTEXT.md]

**Example:**

```text
local process PASS
  -> local PostgreSQL disposable PASS
    -> production read-only migration preflight PASS
      -> human authorizes Heroku release
```

### Pattern 2: Release Phase as migration barrier

**What:** migrations rodam em one-off release dyno antes da nova formação; exit não zero impede a nova release de chegar aos dynos. [CITED: https://devcenter.heroku.com/articles/release-phase]

**When to use:** todo deploy que contém ou pode redescobrir migrations. [CITED: https://devcenter.heroku.com/articles/release-phase]

**Example:**

```text
Procfile release
  -> npm run db:migrate:safe
  -> DATABASE_MIGRATION_URL validated
  -> DTC_RELEASE_MIGRATION_MODE=true
  -> medusa db:migrate
  -> exit 0 permits web/worker boot
```

[VERIFIED: codebase inspection]

### Pattern 3: Dynamic SHA triangulation

**What:** correlacionar o SHA da release Heroku, `health.live.version`, `health.ready.version` e `Sentry.release`. [VERIFIED: codebase inspection]

**When to use:** imediatamente depois do release e no monitor recorrente. [VERIFIED: 12.1-CONTEXT.md]

**Example:**

```text
Heroku release commit SHA
  == /health/live.version
  == /health/ready.version
  == Sentry release (if separately authorized)
```

[CITED: https://devcenter.heroku.com/articles/dyno-metadata; CITED: https://docs.sentry.io/platforms/javascript/guides/node/configuration/releases]

### Pattern 4: Expand/contract-aware rollback

**What:** tratar rollback de código/config e rollback de dados/schema como responsabilidades distintas. [CITED: https://devcenter.heroku.com/articles/releases]

**When to use:** antes de qualquer deploy com migration e antes de registrar o comando de rollback. [CITED: https://devcenter.heroku.com/articles/releases]

**Example:**

```text
rollback Heroku
  -> previous slug/config
  -X-> no automatic rollback of Supabase schema/data
```

### Anti-Patterns to Avoid

- **Promover RC1 histórico a PASS atual:** resultados antigos apenas desenham o gate; não comprovam HEAD, release ou produção atuais. [VERIFIED: artifact scope; inferred only]
- **Fixar o monitor em `vNN`:** a release numérica muda também por config/add-ons; compare SHA factual. [CITED: https://devcenter.heroku.com/articles/releases]
- **Executar migration para “descobrir” o que está pendente:** faça inventário read-only primeiro; `db:migrate` é mutação. [VERIFIED: codebase inspection]
- **Usar `:6543` para migration:** transaction pooling não suporta prepared statements e o script já rejeita essa porta. [VERIFIED: codebase inspection; CITED: https://supabase.com/docs/guides/database/connecting-to-postgres]
- **Interpretar rollback Heroku como rollback de banco:** estado externo não é revertido. [CITED: https://devcenter.heroku.com/articles/releases]
- **Ler ou imprimir config-var values:** nomes bastam para prova de presença; valores violam a política da fase. [VERIFIED: 12.1-CONTEXT.md]
- **Validar provider por mutação real:** ausência de autorização torna isso proibido; declare a limitação. [VERIFIED: 12.1-CONTEXT.md]
- **Confiar em `.medusa/server` já presente:** somente um build fresco do gate autorizado qualifica o artefato. [VERIFIED: repository inspection; inferred operational consequence]

## Common Pitfalls

### Pitfall 1: Gelato aparece novamente como pending

**What goes wrong:** o Release Phase tenta aplicar `Migration20260703000000` mesmo se o DDL Gelato já existe. [VERIFIED: installed package semantics; conditional inference]

**Why it happens:** o histórico pode conter o nome antigo `MigrationTBDGelatoFulfillment`, enquanto o baseline descobre o novo nome pelo arquivo. [ASSUMED]

**How to avoid:** consultar nomes de `public.mikro_orm_migrations` read-only, comparar com as 11 identidades e parar se a identidade Gelato não for a esperada. [VERIFIED: installed package semantics]

**Warning signs:** nome antigo presente, novo ausente, pending set maior que o aprovado ou release output processando Gelato inesperadamente. [VERIFIED: installed package semantics]

### Pitfall 2: Release Phase falha, mas config mudou

**What goes wrong:** a nova formação não sobe, porém uma config var alterada pode permanecer com o valor novo. [CITED: https://devcenter.heroku.com/articles/release-phase]

**Why it happens:** falha do comando impede deploy da release, mas não necessariamente reverte a alteração de config que a criou. [CITED: https://devcenter.heroku.com/articles/release-phase]

**How to avoid:** não misturar mudança de config e deploy neste gate; se inevitável e autorizado, registrar apenas nomes alterados e uma reconciliação específica. [VERIFIED: 12.1 secrets policy; inferred operational recommendation]

**Warning signs:** release failed seguida de comportamento diferente na formação anterior. [CITED: https://devcenter.heroku.com/articles/release-phase]

### Pitfall 3: Rollback deixa schema incompatível

**What goes wrong:** slug antigo volta, mas encontra schema novo e incompatível. [CITED: https://devcenter.heroku.com/articles/releases]

**Why it happens:** rollback Heroku copia slug/config; estado em add-ons/externos não volta. [CITED: https://devcenter.heroku.com/articles/releases]

**How to avoid:** aprovar rollback somente quando migrations do release são backward-compatible ou há plano explícito de reconciliação. [CITED: https://devcenter.heroku.com/articles/releases]

**Warning signs:** drop/rename de coluna/tabela, `down()` destrutivo ou código antigo dependente de contrato removido. [VERIFIED: codebase inspection; inferred operational consequence]

### Pitfall 4: Health saudável com worker ausente

**What goes wrong:** `/health/*` passa no `web.1`, mas jobs/subscribers/workflows não processam. [VERIFIED: codebase architecture]

**Why it happens:** health é servido pelo processo HTTP e não comprova a formação worker. [VERIFIED: codebase architecture]

**How to avoid:** combinar health com `worker.1=up` na mesma janela de evidência. [VERIFIED: 12.1-CONTEXT.md]

**Warning signs:** `web.1` up, `worker.1` down/crashed ou reinícios contínuos. [VERIFIED: 12.1-CONTEXT.md]

### Pitfall 5: Vazamento no pacote de evidência

**What goes wrong:** logs, headers, URLs ou payloads preservam credenciais, PII, client secrets ou Pix. [VERIFIED: project security constraints]

**Why it happens:** captura bruta de CLI/HTTP/provider sem allowlist. [VERIFIED: codebase sanitization design; inferred operational cause]

**How to avoid:** registrar apenas status, nomes de variáveis, códigos de erro, SHA, release e contagens estritamente necessárias; revisar redaction antes de persistir. [VERIFIED: 12.1-CONTEXT.md]

**Warning signs:** `postgres://`, `redis://`, `Bearer`, cookies, `client_secret`, `pi_`, e-mail, telefone, CPF ou QR Pix no artefato. [VERIFIED: sanitize.ts inspection]

### Pitfall 6: Docker aparente, mas indisponível no WSL

**What goes wrong:** o runner escolhe Docker e falha antes do PostgreSQL descartável. [VERIFIED: local availability inspection]

**Why it happens:** o executável Windows está no PATH, mas a integração WSL do Docker Desktop não está ativa. [VERIFIED: local `docker --version` output]

**How to avoid:** o PLAN deve ter checkpoint de disponibilidade antes do gate; fallback permitido é apenas PostgreSQL local realmente descartável via `P12_DISPOSABLE_DATABASE_URL`/`P12_DISPOSABLE_DB_NAME`, nunca Supabase/produção. [VERIFIED: disposable runner inspection]

**Warning signs:** mensagem “docker could not be found in this WSL 2 distro” ou ausência do daemon. [VERIFIED: local availability inspection]

## Objective Rollback Criteria

### No-deploy / stop criteria

Interromper antes do deploy se ocorrer qualquer item abaixo. [VERIFIED: 12.1-CONTEXT.md]

1. baseline não descende de `b4c1...`, diff inclui runtime não aprovado ou lockfile muda; [VERIFIED: 12.1-CONTEXT.md]
2. install, unit, Modules, HTTP, lint ou build falha; [VERIFIED: 12.1 closure criteria]
3. PostgreSQL descartável não é comprovadamente local/descartável ou deixa resíduo; [VERIFIED: Phase 12 harness contract]
4. histórico de migration não corresponde ao inventário aprovado, especialmente Gelato; [VERIFIED: installed migration semantics]
5. `DATABASE_MIGRATION_URL` está ausente/inválida/`:6543`; [VERIFIED: codebase inspection]
6. qualquer config value, URL credenciada ou segredo aparece na evidência. [VERIFIED: 12.1-CONTEXT.md]

### Release failure handling

Se o Release Phase sair não zero, a nova release não entra na formação; registrar status/output sanitizado e classificar BLOCKED. [CITED: https://devcenter.heroku.com/articles/release-phase] Não executar rollback automático nem retry automático; o retry de release só pode ser considerado após causa conhecida e autorização humana. [CITED: https://devcenter.heroku.com/articles/release-phase; VERIFIED: manual gate policy]

### Post-deploy rollback trigger

Abrir o gate humano de rollback se, após uma release bem-sucedida, ocorrer qualquer condição crítica persistente ou reproduzível:

- `web.1` ou `worker.1` não fica ativo; [VERIFIED: 12.1-CONTEXT.md]
- `/health/live` não é `live`; [VERIFIED: 12.1-CONTEXT.md]
- `/health/ready` não é `ready`, PostgreSQL/Redis não estão `up`; [VERIFIED: 12.1-CONTEXT.md]
- health SHA diverge da release realmente implantada; [VERIFIED: 12.1-CONTEXT.md]
- rota essencial retorna 5xx; [VERIFIED: 12.1-CONTEXT.md]
- autenticação administrativa não falha fechada; [VERIFIED: 12.1-CONTEXT.md]
- logs/Sentry mostram regressão crítica em pagamento, criação de Order, fulfillment, segurança ou inicialização; [VERIFIED: blocker policy]
- existe vazamento de segredo/dado relevante. [VERIFIED: blocker policy]

### Proof of rollback readiness without execution

O gate futuro deve identificar a release atual e uma release anterior elegível, registrar o comando exato `heroku releases:rollback <release-anterior> --app espacoliminar`, confirmar que o artefato anterior não expirou e documentar a compatibilidade do schema antes de qualquer execução. [CITED: https://devcenter.heroku.com/articles/heroku-cli-commands; CITED: https://devcenter.heroku.com/articles/releases] Isso prova prontidão operacional, não prova que um rollback foi executado. [VERIFIED: evidence classification policy]

Após rollback autorizado, repetir formação, health, SHA, rotas e logs; não executar `migration:down` automaticamente. [CITED: https://devcenter.heroku.com/articles/releases; VERIFIED: codebase inspection]

### Known Threat Patterns for This Phase

| Pattern | STRIDE | Standard Mitigation |
|---|---|---|
| Credencial exposta em CLI/log/evidence | Information Disclosure | Captura allowlisted, redaction e revisão antes de persistir. [VERIFIED: codebase and context inspection] |
