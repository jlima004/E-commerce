# Phase 12.1: Backend MVP Release Readiness & Production Validation — Context

**Gathered:** 2026-07-24

<domain>
## Phase Boundary

Phase 12.1 is an inserted operational phase for release readiness and production
validation of the merged backend MVP. It must prove, in future separately
authorized gates, that the release candidate can be built, migrated, released,
observed, and rolled back safely on the actual Heroku + Supabase PostgreSQL +
Redis topology.

It adds no product feature, no product requirement, and no storefront work.
Phases 1–12 remain closed. The existing 45/45 product requirements and 56/56
milestone plans remain complete.

### Goal

> Prove that the merged backend MVP can be built, migrated, released and
> observed safely on the actual Heroku + Supabase PostgreSQL + Redis topology,
> with the deployed SHA, process topology and critical backend surfaces
> validated without adding new product features.

### Not an objective

- implement storefront/frontend;
- refactor accepted modules or reopen Phases 1–12;
- correct every historical finding or perform unlimited hardening;
- add alert dashboards, PagerDuty, Slack, SIEM, Correios automation, a provider,
  a payment method, or any other product feature;
- close/archive the milestone automatically.

</domain>

### D12.1-07 — Process topology

- Both must remain active after the release.

### D12.1-08 — Migration

- Runtime connection: pooled where appropriate.
- Migration connection: direct/session connection.
- Release migration: isolated from web/worker runtime.
- Unsafe fallback: prohibited.
- The current release command is migration-only and DB-only; this CONTEXT gate
  does not execute it.

### D12.1-09 — Cache and Redis

Production must preserve:

- Redis cache active;
- Redis event bus active;
- Redis workflow engine active;
- Redis locking active;
- no return to in-memory production defaults.

CACHE-01A/B and INFRA-01 remain closed unless new evidence justifies reopening
them.

### D12.1-10 — Safe production validation

Default future production validation prioritizes:

- health;
- process status;
- version/SHA;
- migration status;
- read-only routes;
- fail-closed authentication;
- sanitized responses;
- absence of 5xx;
- observability;
- logs without secrets.

Any operation that creates a PaymentIntent, Order, refund, exchange, Gelato
order, real email, or real PostHog event requires separate explicit human
authorization.

### D12.1-11 — Real providers

Without separate explicit authorization:

```text
real Stripe mutation: prohibited
real Gelato dispatch: prohibited
real Resend delivery: prohibited
real PostHog delivery: prohibited
real Correios operation: prohibited
```

Provider absence or a provider not exercised must be declared as a limitation;
it cannot be promoted to false PASS evidence.

### D12.1-12 — Rollback

Future Phase 12.1 evidence must produce:

- an identifiable previous release;
- a documented rollback command;
- objective rollback criteria;
- proof that rollback is possible.

Rollback is not executed by this CONTEXT gate.

### D12.1-13 — Health monitor

The monitor contract must validate dynamically:

```text
live = live
ready = ready
postgres = up
redis = up
web.1 = active
worker.1 = active
health SHA = current deployed release SHA
```

The monitor must not be pinned to a specific numeric Heroku release.

### D12.1-14 — Evidence classification

Every future evidence item must be classified as exactly one of:

```text
local process
local PostgreSQL disposable
Heroku release
production read-only
production mutation
real external provider
inferred only
```

Inference must never be promoted to real execution.

### D12.1-15 — Secrets

Never display or persist:

- config-var values;
- full credential-bearing URLs;
- Stripe keys or webhook secrets;
- Gelato API keys;
- Resend or PostHog tokens;
- Redis URLs or `DATABASE_URL`;
- client secrets;
- Pix QR/copy-and-paste values;
- cookies/sessions;
- tracking tokens.

Configuration may be documented by variable name only, never by value.

### Agent discretion

None for scope, evidence promotion, provider mutation, secrets, or gate
progression. Downstream agents may choose only safe implementation details
inside a separately authorized RESEARCH or PLAN gate, while preserving all
decisions above.

## Existing Code Insights

### Reusable assets

- `resolveRuntimeVersion` already centralizes the deployed version used by
  health and Sentry.
- `/health/live` and `/health/ready` already expose the status/version surface
  required for future validation.
- `run-migrations.mjs` already separates `DATABASE_MIGRATION_URL` from runtime
  `DATABASE_URL` and rejects transaction-pooler port 6543.
- `release-migration-mode.ts` already prevents release migrations from running
  as `server` or `worker`.

### Established patterns

- Heroku process topology is defined by `Procfile`: isolated release migration,
  server `web`, and worker-only process.
- Production startup is fail-closed for the required Redis cache, event bus,
  workflow engine, and locking modules.
- Health readiness checks PostgreSQL and Redis; liveness does not perform
  dependency work.
- Version evidence is SHA-based and dynamic, not numeric-release-based.

### Integration points for future gates

- Local release-candidate validation: workspace-scoped Unit, Modules, HTTP,
  lint, and build commands.
- Migration validation: migration inventory plus the isolated release command.
- Production read-only validation: Heroku release/dyno metadata, health routes,
  sanitized logs, and essential public/read-only routes.
- Rollback evidence: factual current and previous Heroku releases, documented
  command, and objective trigger criteria.
