Criar o contrato local planejado para `EmailDeliveryLog`, e-mail de confirmacao, idempotencia, template allowlist e migration draft, sem instalar Resend e sem tocar o fluxo de `Order`.
</objective>

<scope>
Este plano e para execucao futura. Durante o planejamento atual, nao executar este plano.
</scope>

<tasks>

1. Criar modulo `email-delivery-log`
   - type: implementation
   - files:
     - `apps/backend/src/modules/email-delivery-log/index.ts`
     - `apps/backend/src/modules/email-delivery-log/models/email-delivery-log.ts`
     - `apps/backend/src/modules/email-delivery-log/types.ts`
   - action:
     - Definir `EmailDeliveryLog` com `model.define(...)`.
     - Usar prefixo de id como `emlog`.
     - Definir `email_type = order_confirmation`, `template_key = order_confirmation_v1`, `template_version = 1`, `provider = resend`.
     - Definir statuses locais: `recorded`, `queued`, `sending`, `sent`, `failed`, `dead_letter`.
     - Incluir campos de relacao operacional: `order_id`, `cart_id`, `payment_attempt_id`, `checkout_completion_log_id`, `analytics_event_log_id`, `payment_intent_id`.
     - Incluir `idempotency_key`, `recipient_email_hash`, `recipient_email_domain`, `payload`, `metadata`, `provider_message_id`, `attempt_count`, `last_error_code`, `last_error_message`, `next_retry_at`, `recorded_at`, `queued_at`, `sending_started_at`, `sent_at`, `failed_at`, `dead_lettered_at`.
   - acceptance_criteria:
     - Modelo nao importa services de outros modulos.
     - Modelo nao contem campo para e-mail completo do destinatario.
     - Modelo nao contem campo raw provider payload ou Authorization header.

2. Definir indexes/constraints e migration draft
   - type: implementation
   - files:
     - `apps/backend/src/modules/email-delivery-log/models/email-delivery-log.ts`
     - `apps/backend/src/modules/email-delivery-log/migrations/Migration*.ts`
   - action:
     - Criar unique em `email_type + idempotency_key`.
     - Criar guard unique/logico por `email_type + order_id`.
     - Criar indexes por `status + next_retry_at`, `order_id`, `analytics_event_log_id`, `payment_attempt_id`, `checkout_completion_log_id` e `payment_intent_id`.
     - Gerar ou escrever migration draft sem aplicar migration real.
   - acceptance_criteria:
     - `medusa db:migrate` nao e executado neste plano.
     - Migration fica como draft revisavel.

3. Criar helpers puros de idempotency, payload e auditoria
   - type: implementation
   - files:

     - `apps/backend/src/modules/email-delivery-log/types.ts`
   - action:
     - Implementar `buildOrderConfirmationEmailIdempotencyKey({ order_id })`.
     - Retornar `order-confirmation/{order_id}`.
     - Implementar builder allowlist-only de payload/template.
     - Implementar hash do destinatario para auditoria sem persistir e-mail completo.
     - Implementar sanitizacao/rejeicao de metadata proibida.
