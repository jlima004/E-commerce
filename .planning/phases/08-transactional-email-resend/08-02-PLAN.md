# 08-02 - Local Confirmation-Email Enqueue After Order And `purchase_completed`

<objective>
Gravar ou reutilizar `EmailDeliveryLog` localmente depois de `Order` confirmada e depois de `purchase_completed` local duravel, preservando a regra de nascimento da `Order` e sem chamar Resend.
</objective>

<scope>
Este plano e para execucao futura. Durante o planejamento atual, nao executar este plano.
</scope>

<tasks>

1. Registrar o modulo `email-delivery-log` no runtime real do Medusa
   - type: implementation
   - files:
     - `apps/backend/medusa-config.ts`
   - action:
     - Registrar o modulo `email-delivery-log` no runtime real do Medusa.
     - Usar a key sem hifen exigida pelo contrato:

       ```ts
       {
         key: "email_delivery_log",
         resolve: "./src/modules/email-delivery-log",
       }
       ```

     - Se o projeto exigir outra forma de registry em `medusa-config.ts`, documentar a forma real usada no diff do plano/summary futuro e manter `key: "email_delivery_log"` sem hifen.
   - acceptance_criteria:
     - `EmailDeliveryLog` existe no container real do app, nao apenas como injecao/mocking de teste.
     - Build futuro prova que `apps/backend/medusa-config.ts` e o modulo `email-delivery-log` compilam juntos.

2. Integrar o enqueue local ao entrypoint pos-webhook de Order
   - type: implementation
   - files:
     - `apps/backend/src/workflows/order/webhook-order-entrypoint.ts`
     - `apps/backend/src/modules/email-delivery-log/service.ts`
   - action:
     - Resolver o modulo `EmailDeliveryLog` no container.
     - Validar o modulo antes de chamar `completeCartWorkflow` quando possivel.
     - Depois de `Order` confirmada, `CheckoutCompletionLog.completed`, `PaymentAttempt.order_id` correlacionado e `purchase_completed` local duravel, criar/reusar `EmailDeliveryLog`.
     - Usar `order_id` como base da idempotency key `order-confirmation/{order_id}`.
     - Registrar `status = recorded`.
     - Nao chamar Resend.
     - Se o modulo `EmailDeliveryLog` estiver ausente ou mal configurado, retornar erro estavel/sanitizado, nao tratar o webhook/order entrypoint como completamente processado, nao chamar Resend, nao iniciar Gelato e manter o caso elegivel para retry/recovery posterior.
   - acceptance_criteria:
     - `Order` continua nascendo somente pelo entrypoint interno pos-webhook.
     - `purchase_completed` local e pre-condicao para enqueue de e-mail.
     - Replay retorna/reusa e-mail local existente.
     - `EmailDeliveryLog module unavailable -> erro estavel`.
     - Ausencia/mau registro do modulo nunca vira sucesso silencioso.

3. Implementar recovery para Order existente com e-mail local ausente
   - type: implementation
   - files:
     - `apps/backend/src/workflows/order/webhook-order-entrypoint.ts`
     - `apps/backend/src/modules/email-delivery-log/service.ts`
   - action:
     - Nos caminhos `reused_existing_order` / recovery da Phase 06/07, verificar existencia de `purchase_completed` local.
     - Se `purchase_completed` existe e `EmailDeliveryLog` esta ausente, criar e-mail local idempotentemente antes de retornar sucesso do enqueue.
     - Se o e-mail local ja existe em status valido, tratar como reuso.
     - Se `Order` existente + `purchase_completed` existente + `EmailDeliveryLog` ausente + modulo indisponivel, falhar de forma estavel/sanitizada sem sucesso silencioso.
   - acceptance_criteria:
     - `Order` existente + `purchase_completed` existente + e-mail ausente e curada por retry.
     - Duplicidade `email_type + order_id` e tratada como idempotencia ou conflito seguro.
     - Recovery com modulo indisponivel nao retorna sucesso de enqueue, nao chama Resend e nao inicia Gelato.

4. Preservar fonte canonica do destinatario sem persistir e-mail completo
   - type: implementation
   - files:
     - `apps/backend/src/workflows/order/webhook-order-entrypoint.ts`
     - `apps/backend/src/modules/email-delivery-log/service.ts`
   - action:
     - Capturar somente referencias necessarias para o relay resolver `Order.email`.
     - Persistir hash/domain de e-mail somente se aprovado pelo contrato de `08-01`.
     - Nao persistir e-mail completo, Stripe billing details ou payload de request.
   - acceptance_criteria:
     - `EmailDeliveryLog` nao contem e-mail completo.
     - `Order.email` permanece a unica fonte canonica para envio futuro.

5. Criar/atualizar testes unitarios e HTTP
   - type: test
   - files:
     - `apps/backend/src/workflows/order/__tests__/webhook-order-email-enqueue.unit.spec.ts`
     - `apps/backend/integration-tests/http/stripe-webhook-order-creation.spec.ts`
   - action:
     - Cobrir sucesso, replay, concorrencia e recovery.
     - Cobrir ausencia de `purchase_completed` como bloqueio do enqueue local.
     - Cobrir que Resend nao e chamado neste plano.
     - Adicionar teste unitario obrigatorio: `EmailDeliveryLog module missing or misconfigured -> no silent success`.
     - O teste deve cobrir que nao chama Resend, nao inicia Gelato, nao retorna sucesso de enqueue, retorna erro estavel/sanitizado e, se o modulo for validado antes de `completeCartWorkflow`, nao chama `completeCartWorkflow`.
   - acceptance_criteria:
     - Uma `Order` confirmada com `purchase_completed` gera exatamente um `EmailDeliveryLog`.
     - Replays/concurrency nao duplicam `Order`, analytics event ou email log.
     - Resend nao participa deste plano.
     - Modulo `EmailDeliveryLog` ausente/mal configurado e fail-closed, sem sucesso silencioso.
