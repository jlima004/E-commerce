# 10-02 - Public Token-Gated Tracking Route With Sanitized Response

## Objective

Expose a public tracking lookup route that works only with a valid
`TrackingAccessToken` and returns a minimal sanitized tracking response derived
from local Order/Gelato state.

## Route Contract

Use:

```http
POST /store/tracking/lookup
Content-Type: application/json
```

Request body:

```json
{ "token": "one-time-visible-token-value" }
```

Do not expose token in path or query string.

## Response Contract

Allowed fields:

- safe public order reference, if available;
- order status summary;
- fulfillment status summary;
- tracking status summary;
- item count or already-public item labels;
- `updated_at`;
- safe message when tracking is not available yet.

Forbidden fields:

- submitted token;
- token hash;
- internal `order_id` if it is not already a safe public reference;
- full e-mail;
- phone;
- CPF/CNPJ;
- full address or address lines;
- payment data;
- Stripe client secret;
- Pix QR/copia-e-cola;
- raw Order payload;
- raw Gelato payload;
- `trackingCode`;
- `trackingUrl`;
- headers;
- cookies;
- secrets.

## Tasks

1. Add route file for `POST /store/tracking/lookup`.
2. Validate body with a strict schema: only `token` accepted.
3. Reject requests containing `order_id`, e-mail, phone, CPF/CNPJ, address,
   payment IDs, or extra lookup identifiers.
4. Resolve `TrackingAccessToken` by hash and constant-time comparison.
5. Reject invalid, unknown, expired, revoked, or malformed token with the same
   safe public error shape.
6. Resolve local Order/Gelato fulfillment state without exposing raw module
   payloads.
7. Add serializer that allowlists only the public response fields.
8. Ensure access log route template cannot include the token.
9. Add HTTP integration tests for valid token, invalid token, expired token,
   revoked token, and no-token cases.
10. Add negative tests proving `order_id`, e-mail, phone, CPF/CNPJ and address
    lookup are not accepted.

## Verification

Future execution should run targeted route tests plus focused unit tests from
10-01 as needed.

Do not run real Gelato, webhook smoke, refund, exchange, real migration, deploy,
or Phase 11.

## Acceptance Criteria

- Guest can retrieve sanitized tracking status with a valid token.
- No lookup works by `order_id` alone.
- No lookup works by e-mail, phone, CPF/CNPJ, address, cart id, or payment id.
- Public response excludes all sensitive fields.
- Token is not returned in response.
- Token is not placed in route path/query.
- Phase 11 remains unstarted.
