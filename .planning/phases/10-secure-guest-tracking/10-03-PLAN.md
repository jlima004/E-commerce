# 10-03 - Rate Limit, Enumeration Protection And Final Validation

## Objective

Harden the public tracking route against replay/enumeration and complete Phase
10 validation with documented negative proofs. This is the final slice before
manual review.

## Scope

Future execution may add:

- rate-limit helper/model inside the tracking module if needed;
- non-PII rate bucket hashing;
- per-window counters for invalid/malformed attempts;
- public 429 response with no token existence disclosure;
- final validation documentation in `10-03-SUMMARY.md`.

## Rate Limit Contract

Rate-limit buckets must not persist raw IP or full user-agent.

Preferred bucket:

- HMAC of client IP + summarized user agent + time window;
- short TTL or window reset;
- sanitized status/counter fields only.

If a token row exists, per-token failure counters may be added, but the public
response must not reveal whether the row exists.

## Tasks

1. Add rate-limit/enumeration guard around `POST /store/tracking/lookup`.
2. Ensure malformed and invalid requests count toward the same public lookup
   limit.
3. Ensure rate-limited responses do not disclose token existence.
4. Ensure invalid, unknown, expired, and revoked tokens have indistinguishable
   public error body.
5. Add tests for repeated invalid token attempts.
6. Add tests proving raw IP and full user-agent are not persisted/logged.
7. Add tests proving token values do not appear in logs/Sentry contexts.
8. Run the approved Phase 10 validation battery.
9. Document results and negative proofs in `10-03-SUMMARY.md`.

## Required Negative Proofs

Document proof that:

- plaintext token is never persisted;
- token hash comparison is constant-time;
- invalid token is rejected;
- expired token is rejected;
- revoked token is rejected;
- `order_id` without token does not work;
- e-mail lookup does not work;
- phone lookup does not work;
- CPF/CNPJ lookup does not work;
- public response does not contain full address, full e-mail, phone, CPF/CNPJ,
  payment data, secrets, headers, or full Gelato payload;
- tracking route does not allow enumeration;
- logs and Sentry are sanitized;
- no refund/exchange/admin ops are introduced;
- Phase 11 is not started.

Use scoped negative greps for the runtime files touched by Phase 10 and broad
informational greps for repository-wide awareness. Document any false positives
explicitly.

## Acceptance Criteria

- Public tracking is token-gated and rate-limited.
- Enumeration/replay attempts are constrained.
- Sensitive fields are not exposed.
- Token plaintext is never persisted or logged.
- TRK-01 and TRK-02 are ready for manual review.
- No real Gelato, webhook smoke, refund, exchange, migration application,
  deploy, or Phase 11 work is introduced.
