## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| storefront futura -> Store API cart ativo | Request nao confiavel tenta escolher customer/cart ativo. |
| Store API -> Medusa core cart | Mutacao de cart pre-Order deve permanecer sem payment/order. |
| Store API -> logs/Sentry | Request pode conter email/endereco/documento em fases posteriores; logs continuam allowlist. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-S | Spoofing | `resolveActiveCartIdentity` | mitigate | Derivar customer de autenticacao Medusa e guest de sessao/cookie/header assinado existente; nunca aceitar `customer_id` arbitrario. |
| T-03-01-T | Tampering | `active-cart.ts` metadata de cart | mitigate | Marcar cart substituido apenas por metadata/core existente; se exigir schema novo, gate manual antes de qualquer migration. |
| T-03-01-I | Information Disclosure | access log/Sentry | mitigate | Reusar allowlist da Phase 01; nao logar body/query, cart payload amplo, email, endereco ou documento. |
| T-03-01-E | Elevation of Privilege | `GET/POST /store/carts/active` | mitigate | Customer autenticado so opera cart associado; guest so opera cart provado como da sessao atual. |
| T-03-01-B | Tampering | checkout boundary | mitigate | Grep e testes impedem completion/payment/order/webhook/fulfillment neste plano. |
</threat_model>
