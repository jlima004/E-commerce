<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cliente autenticado -> rota de attach | Cliente pode tentar anexar cart id de outra sessao. |
| rota de attach -> Medusa transfer cart | Transferencia altera ownership do cart. |
| rota de attach -> logs/Sentry | Falhas de autorizacao podem conter IDs e PII se mal formatadas. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02-S | Spoofing | `resolveCurrentSessionGuestCart` | mitigate | Exigir prova server-side de sessao atual; body sozinho nao autoriza attach. |
| T-03-02-T | Tampering | `shouldTransferGuestCart` | mitigate | Transferir somente cart nao vazio; preservar customer cart quando guest esta ausente/vazio. |
| T-03-02-R | Repudiation | rota de attach | mitigate | Registrar apenas outcome e correlation_id, sem body, IDs sensiveis ou documento. |
| T-03-02-I | Information Disclosure | erros de attach | mitigate | Respostas saneadas nao revelam se cart de terceiro existe. |
| T-03-02-E | Elevation of Privilege | transferencia de cart | mitigate | Customer derivado da autenticacao; guest cart provado por sessao atual; nenhum `customer_id` aceito do body. |
| T-03-02-B | Tampering | checkout boundary | mitigate | Grep e testes impedem payment/order/webhook/fulfillment no attach. |
</threat_model>
