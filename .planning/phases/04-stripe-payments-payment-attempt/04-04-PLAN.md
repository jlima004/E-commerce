## Phase Goal

Permitir que o cliente inicie cartao Stripe em BRL a partir de um cart elegivel, recebendo somente os dados imediatos necessarios para confirmar no cliente, enquanto qualquer persistencia local/Medusa fica limitada a allowlist segura.

## Post-Gate Binding

04-01 substitui a hipotese native-first pura:

```
NATIVE_CARD_SAFE=false
CUSTOM_PROVIDER_OR_LAYER_REQUIRED=true
PAYMENTSESSION_SECRET_PERSISTENCE_BLOCKER=true
BLOCK_NATIVE_IF_SECRET_PERSISTED=true
```

Este plano so pode executar runtime de card depois que a fronteira segura provar uma das estrategias abaixo:

- custom Stripe provider que retorna `PaymentSession.data` allowlist-only;
- camada propria Stripe que nao grava nem reexpoe `PaymentSession.data` bruto;
- wrapper/camada de filtragem que intercepta antes da persistencia e da exposicao Store API.

Se nenhuma estrategia provar `PaymentSession.data` allowlist-only, parar sem implementar rota/helper card.

## Artifacts this phase produces

- `SafeStripePaymentData` / `SafeStripeImmediateCardAction` ou nomes equivalentes.
- `toSafeStripePaymentSessionData` ou boundary equivalente allowlist-only.
- `POST /store/carts/:id/payment-attempts/card`.
- `startCardPaymentAttempt`.
- `CardPaymentAttemptResponse`.
- `markCardClientConfirmed` ou equivalente local, sem verdade financeira.
- Unit tests de boundary provando que PaymentIntent bruto, `client_secret` e `next_action` nao entram em persistencia.
- Integration tests HTTP para cartao sem Order.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser Stripe.js -> backend | Backend nao recebe dados brutos de cartao. |
| Stripe/provider response -> persistence | `client_secret` e PaymentIntent bruto devem ser separados do shape persistivel antes de qualquer persistencia/log. |
| PaymentSession.data -> Store API | Se existir, `data` precisa ser allowlist-only e nunca reexpor payload Stripe bruto. |
| Client confirmation -> financial truth | Confirmacao cliente nao equivale a pagamento aprovado. |
