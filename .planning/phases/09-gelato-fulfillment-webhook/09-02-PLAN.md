# 09-02 - Local Gelato Eligibility Gate After Order, purchase_completed And Email Sent

## Objective

Add the local eligibility contract and fail-closed runtime module registration needed for Gelato fulfillment after the accepted downstream prerequisites exist: confirmed `Order`, durable local `purchase_completed`, and `EmailDeliveryLog(order_confirmation).status = sent`.

## Scope

This slice may create eligibility helpers and may create/reuse local fulfillment only in replay/recovery paths where e-mail is already `sent`. It must not depend on a future Stripe webhook replay after the e-mail becomes `sent`; the regular post-email path is handled by the `09-03` relay eligibility scan. It must not call Gelato. It must not implement operational override, tracking public routes, refund, exchange, Stripe CLI smoke, or Phase 10 behavior.

Register the module in the real Medusa runtime:

```ts
{
  key: "gelato_fulfillment",
  resolve: "./src/modules/gelato-fulfillment",
}
```

If the repository's real Medusa module pattern requires a different surrounding shape, follow that pattern, but keep the key without hyphen: `gelato_fulfillment`.

## Eligibility Contract

Automatic Gelato is eligible only when all are true:

- `Order` exists through the canonical post-webhook path.
- `Order.metadata.order_status = confirmed` and `payment_status = captured` or equivalent accepted local state from Phase 06.
- `AnalyticsEventLog(event_name = purchase_completed, order_id = order.id)` exists locally in any durable local status accepted by Phase 07.
- `EmailDeliveryLog(email_type = order_confirmation, order_id = order.id).status = sent`.
- No active/sent/dead-letter fulfillment already exists for the `Order`.

Explicitly false:

- PostHog success is not checked.
- `AnalyticsEventLog.status = sent` is not checked.
- `EmailDeliveryLog.status = recorded|queued|sending|failed|dead_letter` blocks automatic Gelato.
- Email `dead_letter` does not become an override.
- Operational override is reserved only as future contract, not implemented.

## Tasks

1. Add pure eligibility helper that accepts projected `Order`, `AnalyticsEventLog`, `EmailDeliveryLog`, and existing fulfillment records.
2. Add local record/claim helper for a `GelatoFulfillment` in `eligible` or `queued` status.
3. Register the module `gelato-fulfillment` in the real Medusa runtime config using key `gelato_fulfillment`.
4. Integrate only as an Order-entrypoint recovery/replay check without changing Order birth behavior and without relying on a future Stripe webhook replay after e-mail delivery.
5. Ensure missing/misconfigured Gelato module fails closed after Order/log creation rather than reverting `Order`.
6. Add recovery/replay behavior for existing Order + purchase_completed + email sent.
7. Add unit and filtered HTTP tests for blocked/not-blocked cases.

## Verification

Future execution must prove:

- No Gelato client is constructed or called in this slice.
- `dead_letter` email blocks automatic Gelato.
- Missing `purchase_completed` blocks Gelato.
- PostHog relay state is irrelevant.
- Replay/concurrency produces exactly one local fulfillment claim.
- `GelatoFulfillment module missing or misconfigured -> no silent success`:
  - does not call Gelato;
  - does not create external fulfillment;
  - does not return silent success;
  - emits a stable/sanitized error;
  - does not revert already persisted `Order`, `purchase_completed`, or `EmailDeliveryLog`;
  - leaves the case recoverable by retry/recovery.
