# 09-04 - Gelato Webhook Ingestion, Deduplication, Status And Tracking Update Contract

## Objective

Implement Gelato webhook ingestion with fail-closed authenticity, `WebhookEventLog` persistence/deduplication, and idempotent fulfillment status/tracking updates.

## Scope

Do not expose public tracking. Do not implement refund/exchange. Do not run real webhook smoke. Do not call Gelato real. This slice must start by resolving the authenticity blocker in `09-RESEARCH.md`.

- `D09-11`

## Pre-Implementation Blocker Gate

Before implementing a public webhook route, refresh official Gelato docs or account/API portal guidance for one of:

- HMAC/signature header + webhook secret;
- documented app-level secret/header;
- IP allowlist guidance plus secret route/token;
- another official authenticity mechanism.

If no official authenticity mechanism is confirmed, stop this slice with `09-04-SUMMARY.md` marked blocked. Do not create an accepting unauthenticated webhook route.

## Webhook Contract

Route target planned:

```text
POST /hooks/gelato
```

Accepted event:

```text
order_status_updated
```

Planned dedupe:

- `provider = gelato`
- `external_event_id = payload.id`
- `deduplication_key = payload.id` when present
- `payload_hash` stored for audit/fallback
- `entity_type = fulfillment`
- `entity_id = GelatoFulfillment.id` when resolved

Update lookup:

- Resolve fulfillment by `orderReferenceId = order_id`.
- Match/attach `orderId` against `gelato_primary_order_id` or `connected_order_ids`.
- For split orders, aggregate item status/tracking into same local fulfillment.

Tracking contract:

- Persist tracking only inside fulfillment local summary.
- Do not create public token, public route, public tracking controller or Phase 10 behavior.

## Tasks

1. Confirm authenticity scheme or stop blocked.

3. Add route-level validation for event type and authenticity before any effect.

6. Add replay/concurrency tests for duplicate webhook id.
7. Add tests for split-order connected webhook updates.
8. Add negative tests for invalid authenticity, unknown fulfillment and forbidden payload data.

## Verification

Future execution must prove:

- forged/unsigned webhook is rejected before DB side effects;
- duplicate event id is no-op;
- status/tracking update is idempotent;
- tracking remains non-public;
- raw body/full payload/headers/secrets are not persisted.

- No unauthenticated accepting public route exists.
