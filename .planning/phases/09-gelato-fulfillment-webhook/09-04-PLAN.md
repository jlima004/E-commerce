# 09-04 - Gelato Webhook Ingestion, Deduplication, Status And Tracking Update Contract

## Objective

Implement Gelato webhook ingestion with fail-closed HTTP Header authenticity, `WebhookEventLog` persistence/deduplication, and idempotent fulfillment status/tracking updates.

## Scope

Do not expose public tracking. Do not implement refund/exchange. Do not run real webhook smoke. Do not call Gelato real. Do not run Stripe CLI smoke. Do not start Phase 10.

Webhook authenticity is **confirmed documentally** via Gelato dashboard/API Portal (Authorization Type = HTTP Header). Implementation must verify fail-closed before any DB side effect.

## Pre-Implementation Auth Gate (Resolved Documentally)

Gelato dashboard/API Portal confirms webhook auth via **HTTP Header**:

- Authorization checkbox
- Authorization Type: HTTP Header
- Header Name (configurable)
- Header Value (configurable)

No HMAC/signature/timestamp confirmed. Do not reuse `GELATO_API_KEY` as webhook secret.

Chosen mechanism for Phase 09:

- Dedicated header: `X-GELATO-WEBHOOK-SECRET`
- Env: `GELATO_WEBHOOK_AUTH_HEADER_NAME` (expected default: `X-GELATO-WEBHOOK-SECRET`)
- Env: `GELATO_WEBHOOK_SECRET` (shared secret configured in Gelato dashboard)

Implementation contract:

- Reject before any DB side effect if header missing or incorrect (fail-closed).
- Replay/dedupe via `WebhookEventLog` using `payload.id`; `payload_hash` only as safe fallback when `id` absent.
- Do not create an accepting unauthenticated webhook route.

## Webhook Contract

Route target planned:

```text
POST /hooks/gelato
```

Official Gelato event names use underscores. Phase 09 MVP accepts **only**:

```text
order_status_updated
```

Out of MVP (future decision required):

- `order_item_status_updated`
- `order_item_tracking_code_updated`
- `order_delivery_estimate_updated`
- `store_product_template_created`
- `store_product_template_updated`
- `store_product_template_deleted`
- `store_product_created`
- `store_product_updated`
- `store_product_deleted`

Planned dedupe:

- `provider = gelato`
- `external_event_id = payload.id`
- `deduplication_key = payload.id` when present
- `payload_hash` stored for audit/fallback only when `id` absent and safe to persist/ignore without duplicate effects
- `entity_type = fulfillment`
- `entity_id = GelatoFulfillment.id` when resolved

Update lookup:

- Resolve fulfillment by `orderReferenceId = order_id`.
- Match/attach `orderId` against `gelato_primary_order_id` or `connected_order_ids`.
- For split orders, aggregate item status/tracking into same local fulfillment.

Tracking contract:

- Persist tracking only inside fulfillment local summary.
- Do not create public token, public route, public tracking controller or Phase 10 behavior.
