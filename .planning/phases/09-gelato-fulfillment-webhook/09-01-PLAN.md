# 09-01 - Gelato Fulfillment Contract, Local Model, Idempotency And Single-Active Guard

## Objective

Create the local Gelato fulfillment contract without dispatching anything to Gelato. This slice defines the audit model, idempotency key, status lifecycle, sanitized payload summary, and single-active guard per `Order`.

## Scope

Implement only local module/model/helper/test work. Do not call Gelato, do not register a real relay, do not execute migrations, do not alter checkout/Stripe/Order birth behavior, and do not start webhook ingestion.

## Planned Contract

Plan a `GelatoFulfillment` aggregate with at least:

- `id`
- `order_id`
- `cart_id`
- `payment_attempt_id`
- `checkout_completion_log_id`
- `analytics_event_log_id`
- `email_delivery_log_id`
- `idempotency_key` = `gelato-dispatch:{order_id}`
- `order_reference_id` = local `order_id`
- `customer_reference_id`
- `status`
- `gelato_primary_order_id`
- `connected_order_ids`
- `request_hash`
- `request_summary`
- `response_summary`
- `tracking_summary`
- `attempt_count`
- `last_error_code`
- `last_error_message`
- `next_retry_at`
- `requires_operator_attention`
- `operator_alert_code`
- `operator_alert_message`
- `operator_alerted_at`
- timestamps for recorded/queued/dispatching/submitted/accepted/failed/dead_lettered

Status set should cover local and Gelato lifecycle without exposing Phase 10 tracking:

```text
recorded
eligible
queued
dispatching
submitted
accepted
in_production
partially_shipped
shipped
delivered
failed
dead_letter
canceled
```

Single-active rule:

- Prefer unique `order_id` for MVP to prevent any second fulfillment row for the same `Order`.
- Treat `connected_order_ids` as internal Gelato split parts of the same fulfillment.
- Future reprocess/cancel/recreate requires explicit operational decision outside this slice.

## Tasks

1. Add module skeleton for `gelato-fulfillment` following local Medusa module patterns.
2. Add model draft and migration draft only; do not run `medusa db:migrate`.
3. Add type constants for statuses, provider, metadata, request/response summaries and tracking summary.
4. Add helper `buildGelatoDispatchIdempotencyKey({ order_id })`.
5. Add helper to classify active vs terminal statuses.
6. Add guard helper that rejects a second active fulfillment for the same `order_id`.
7. Add sanitizers that allow only audit-safe metadata and summaries.
8. Add minimal operator-alert fields on the fulfillment aggregate for `FUL-04`; do not create the broad Phase 12 `OperationalAlert` module.
9. Add unit tests for idempotency, status validation, single-active logic, `connectedOrderIds` aggregation, operator-alert field sanitization and forbidden data rejection.

## Verification

Future execution must run targeted unit tests only for this slice. Build is required if module registration/runtime config is touched in this slice; otherwise defer full build to later runtime slice.

## Acceptance Criteria

- Local fulfillment contract exists and is covered by unit tests.
- `gelato-dispatch:{order_id}` is the only local dispatch idempotency key.
- A single `Order` cannot have more than one fulfillment aggregate.
- `connectedOrderIds` do not create extra local fulfillments.
- No Gelato real call, webhook route, tracking public route, refund, exchange, Stripe CLI smoke, runtime migration or Phase 10 work is introduced.
