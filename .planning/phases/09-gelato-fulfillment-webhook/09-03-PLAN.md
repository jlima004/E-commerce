# 09-03 - Async Gelato Dispatch Relay With Eligibility Scan, Retry, Backoff And Dead Letter

## Objective

Implement the asynchronous Gelato dispatch relay using an eligibility scan, fake/injectable client in tests, local claim semantics, retry/backoff, and persistent failure handling. This is the first slice allowed to define the outbound Gelato client, but tests must not call Gelato real.

## Scope

No SDK install. Use Node `fetch` or a thin injectable client. No real Gelato order creation. No runtime migration execution. No webhook ingestion in this slice.

## Dispatch Contract

Before each dispatch batch, the relay executes eligibility scan + create/reuse local `GelatoFulfillment` before dispatch:

- find confirmed Orders;
- require durable local `purchase_completed`;
- require `EmailDeliveryLog(order_confirmation).status = sent`;
- create or reuse a local `GelatoFulfillment`;
- then perform claim/dispatch asynchronously.

Required mechanism: Eligibility scan + create/reuse local GelatoFulfillment before dispatch.

This scan is the normal mechanism for the case where `EmailDeliveryLog` turns `sent` after the original Stripe webhook/Order flow. It must not require Stripe webhook replay.

Outbound request:

- `POST https://order.gelatoapis.com/v4/orders`
- Header `Content-Type: application/json`
- Header `X-API-KEY`, never logged/persisted
- `orderType = order`
- `orderReferenceId = order_id`
- `customerReferenceId` derived from customer/order context without exposing sensitive data in logs
- `currency = BRL`
- `items[]` from immutable line-item `gelato_snapshot`
- `shippingAddress` from accepted checkout/order data, including BR `federalTaxId` only transiently when required
- `metadata` limited to safe short references, never raw payload/sensitive fields

Local persistence after response:

- Store Gelato primary `id`.
- Store `connectedOrderIds` as same fulfillment aggregate.
- Store fulfillment/financial status summary.
- Store request hash and sanitized request/response summaries only.

Retry:

- Retry `429` and `5xx` with exponential backoff.
- Treat `400`, `401`, `404` as non-transient by default.
- Persistent failure becomes `dead_letter` and marks the local minimal operator alert fields on `GelatoFulfillment`: `requires_operator_attention = true`, `operator_alert_code`, `operator_alert_message`, and `operator_alerted_at`. Do not create the broad Phase 12 `OperationalAlert` module in this phase.

## Tasks

1. Add env/config helpers for Gelato relay enablement and API key without adding dependencies.
2. Add injectable `GelatoDispatchClient` interface and default `fetch` implementation.
3. Add payload builder from immutable Order line items and accepted shipping data.
4. Add relay job following `email-resend-relay` style: eligibility scan, create/reuse local fulfillment, list due candidates, claim, dispatch, success/failure update.
5. Add fake-client unit tests for success, split order response, transient failure, permanent failure, missing config, disabled config and payload sanitization.
6. Add tests proving `EmailDeliveryLog sent after original Order webhook -> relay creates/reuses GelatoFulfillment without requiring Stripe webhook replay`.
7. Add tests proving `EmailDeliveryLog dead_letter -> relay does not create GelatoFulfillment`.
8. Add tests proving `EmailDeliveryLog recorded|queued|sending|failed -> relay does not create GelatoFulfillment`.
9. Add proof that Gelato failure does not revert Order or local downstream logs.

- Tests use fake Gelato client only.
- Split order response maps to one fulfillment aggregate.
- Eligibility scan creates/reuses local `GelatoFulfillment` after e-mail `sent` without Stripe webhook replay.
- Persistent failures surface as `dead_letter` plus local `GelatoFulfillment` operator-alert fields.
- No refund, exchange, tracking public, Stripe CLI smoke or Phase 10 work is introduced.
