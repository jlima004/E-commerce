# 11-02 - Stripe Refund Webhook Confirmation

## Objective

Make Stripe refund webhook confirmation the only source of local final refund
truth. Recompute local financial state transactionally while preserving the
decoupled operational `order_status`.

## Scope

Future execution may add:

- refund-event handling inside the existing Stripe webhook path;
- `WebhookEventLog` dedupe/replay behavior for refund events;
- correlation from Stripe refund object to `RefundRequest`;
- terminal success/failure state transitions;
- transactional confirmed-refund amount recomputation;
- `payment_status` recomputation for partial and total refunds;
- focused webhook and concurrency tests.

## Tasks

1. Extend the existing Stripe webhook processor for refund-related events.
2. Accept only signature-verified, deduped events through the existing webhook
   ingestion boundary.
3. Correlate by `stripe_refund_id`, `payment_intent`, and local request data.
4. Treat refund object events as the canonical source for final local financial
   truth; treat non-terminal refund states as not final financial truth.
5. On confirmed success, transactionally:
   - mark the `RefundRequest` confirmed;
   - recompute confirmed refunded amount;
   - recompute `payment_status`;
   - preserve `order_status`.
6. Ensure `charge.refunded` cannot duplicate accounting with `refund.updated`;
   if processed, it must be informational/idempotent and subordinate to the
   refund object.
7. On failed/canceled terminal state, mark the request failed/canceled and do
   not subtract confirmed refunded amount.
8. Ensure duplicate and concurrent webhook deliveries are no-ops after first
   accepted processing.
9. Add tests proving partial refund, total refund, failed refund, replay, and
   concurrency behavior.

## Verification

Future execution should run focused webhook tests and any build required by
runtime registration changes.

Do not run Stripe CLI smoke. Do not call Stripe real. Do not apply real
migration. Do not create exchange workflow. Do not start Phase 12.

## Acceptance Criteria

- Local final refund truth updates only after trusted Stripe webhook
  confirmation from refund object events.
- `refund.created` / non-terminal states do not finalize local financial state.
- `charge.refunded` does not duplicate `confirmed_refunded_amount` when paired
  with `refund.updated`.
- Partial refund recomputes financial state correctly.
- Total refund recomputes financial state correctly.
- Refund does not automatically set `order_status = canceled`.
- Total refund does not automatically set `order_status = canceled`.
- Replay/concurrency cannot exceed captured amount.
- No Order birth rule change, Gelato change, Correios integration, Stripe CLI
  smoke, real migration, deploy, or Phase 12 work is introduced.
