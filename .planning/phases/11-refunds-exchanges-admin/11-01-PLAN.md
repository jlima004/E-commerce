# 11-01 - RefundRequest Contract/Model/Admin-Safe Command

## Objective

Create the local Admin refund request contract and amount guard without marking
final local financial state as refunded. The request path may reserve local
amount safely and may prepare an injectable Stripe boundary, but final money
truth remains blocked until `11-02` webhook confirmation.

## Scope

Future execution may add:

- `refund-request` module/model/service;
- draft migration only, unless real migration is separately approved;
- Admin-only refund request route/command;
- idempotency key handling;
- amount/currency/captured-availability guard;
- pending reservation semantics for non-terminal refund requests;
- fake/injectable Stripe refund creation boundary;
- focused unit and Admin/API tests.

## Tasks

1. Create `RefundRequest` model and status vocabulary.
2. Add draft migration for the model; do not apply it.
3. Register the module only if this slice owns the runtime registration and then
   run the required future build gate.
4. Implement helper to compute refundable availability from local captured
   payment truth minus confirmed refunds and non-terminal reservations.
   Stop before runtime implementation if the canonical local source for
   `captured_amount`, currency, and payment reference is not clearly identified.
5. Reject zero, negative, over-captured, and currency-mismatched requests.
6. Require an existing Order created by the accepted Phase 06 flow.
7. Add idempotency key reuse for repeated Admin requests.
8. Create an Admin-safe command/route that records request/reservation state.
9. Add an injectable Stripe boundary/fake client if Stripe refund creation is
   included in this slice; never call Stripe real in tests.
10. Keep `payment_status` and confirmed refunded amount unchanged.
11. Add tests for validation, idempotency, concurrency guard, and no Order
    creation.

## Verification

Future execution should run focused tests for this slice. If the slice changes
module registration, `env.ts`, or `medusa-config.ts`, it must also run:

```bash
cd apps/backend && HOME=/tmp XDG_CONFIG_HOME=/tmp TMPDIR=/tmp ADMIN_DISABLED=true npm run build
```

Do not run real migration. Do not mark refund financial truth as confirmed. Do
not implement exchange workflow. Do not start Phase 12. If canonical captured
amount/currency/payment-reference evidence is unclear, stop at the manual gate
before runtime code.

## Acceptance Criteria

- Operator/Admin path can request a refund through local contract.
- Zero/negative refund is rejected.
- Refund above captured availability is rejected.
- Currency mismatch is rejected.
- Concurrent/idempotent requests cannot over-reserve captured amount.
- Refund request never creates Order.
- Refund request never changes `order_status`.
- Local final financial state is not updated until `11-02`.
- No real Stripe, Gelato, Correios, migration, deploy, Stripe CLI smoke, or
  Phase 12 work is introduced.
