# 07-03 - Async PostHog Relay, Retry And Final Gate

<objective>
Implementar o relay assincrono de `AnalyticsEventLog` para PostHog sem transformar PostHog em gate de negocio, e fechar a Phase 07 com provas negativas contra Email, Gelato, fulfillment, refund e tracking.
</objective>

<scope>
Este plano e para execucao futura. Durante o planejamento atual, nao executar este plano, nao instalar dependencia e nao chamar PostHog real.
</scope>

<tasks>

1. Adicionar dependencia/configuracao PostHog somente para relay
   - type: implementation
   - files:
     - `apps/backend/package.json`
     - `apps/backend/package-lock.json` if it exists in the execution checkout
     - arquivos de configuracao necessarios sob `apps/backend/src/jobs` ou loader local
   - action:
     - Adicionar SDK PostHog Node conforme versao aprovada no stack da fase.
     - Se `apps/backend/package-lock.json` existir, atualizar o lockfile de forma controlada junto com a instalacao do SDK.
     - Ler envs do relay de forma fail-closed para entrega externa: token/host ausentes impedem apenas o envio, nao o registro local.
     - Nao adicionar env obrigatoria ao caminho de Order creation.
   - acceptance_criteria:
     - Lockfile existente permanece consistente com `package.json`.
     - Sem chamada real PostHog em teste.
     - Ausencia de config PostHog nao impede `AnalyticsEventLog.recorded`.

2. Criar scheduled job/worker de relay
   - type: implementation
   - files:
     - `apps/backend/src/jobs/analytics-posthog-relay.ts`
     - `apps/backend/src/modules/analytics-event-log/service.ts`
   - action:
     - Selecionar eventos `recorded` e `failed` com `next_retry_at <= now`.
     - Marcar evento como `queued`/`sending` antes da chamada externa.
     - Chamar `client.capture({ event, distinctId, properties })` com payload allowlist-only.
     - Usar `order_id` como `distinctId` MVP para evitar PII.
     - Chamar flush/shutdown do cliente ao fim do job quando aplicavel.
   - acceptance_criteria:
     - Job nao altera `Order`, `PaymentAttempt` ou `CheckoutCompletionLog`.
     - Job nao chama Email, Gelato, refund ou tracking.

3. Implementar retry/backoff e dead letter
   - type: implementation
   - files:
     - `apps/backend/src/modules/analytics-event-log/service.ts`
     - `apps/backend/src/jobs/analytics-posthog-relay.ts`
   - action:
     - Em sucesso: marcar `sent` e `sent_at`.
     - Em falha transiente: marcar `failed`, incrementar `attempt_count`, salvar erro sanitizado e definir `next_retry_at`.
     - Em limite de tentativas: marcar `dead_letter`.
     - Garantir que `failed` e `dead_letter` ainda satisfazem o gate local para downstream.
   - acceptance_criteria:
     - Falha PostHog nao reverte Order.
     - Falha PostHog nao apaga evento local.
     - Falha PostHog nao bloqueia downstream que depende de evento local.

4. Completar validacao final da Phase 07
   - type: test
   - files:
     - `apps/backend/src/jobs/__tests__/analytics-posthog-relay.unit.spec.ts`
     - `apps/backend/src/workflows/order/__tests__/webhook-order-analytics-outbox.unit.spec.ts`
     - `apps/backend/integration-tests/http/stripe-webhook-order-creation.spec.ts`
   - action:
     - Cobrir sucesso do relay com fake client.
     - Cobrir PostHog indisponivel.
     - Cobrir `failed`/`dead_letter` como gate local ainda valido.
     - Rodar unit, HTTP, build e greps negativos.
   - acceptance_criteria:
     - ANL-01, ANL-02 e ANL-03 cobertos.
     - `07-03-SUMMARY.md` para no manual gate antes das fases 08/09.

</tasks>
