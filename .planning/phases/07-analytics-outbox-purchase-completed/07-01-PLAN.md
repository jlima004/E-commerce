# 07-01 - AnalyticsEventLog Contract, Model And Idempotency

<objective>
Criar o contrato local planejado para `AnalyticsEventLog` e `purchase_completed`, sem acoplar PostHog e sem tocar o entrypoint de criacao de `Order`.
</objective>

<scope>
Este plano e para execucao futura. Durante o planejamento atual, nao executar este plano.
</scope>

<tasks>

1. Criar modulo `analytics-event-log`
   - type: implementation
   - files:
     - `apps/backend/src/modules/analytics-event-log/index.ts`
     - `apps/backend/src/modules/analytics-event-log/models/analytics-event-log.ts`
     - `apps/backend/src/modules/analytics-event-log/types.ts`
   - action:
     - Definir `AnalyticsEventLog` com `model.define(...)`.
     - Usar prefixo de id como `anlevt`.
     - Definir `event_name = purchase_completed`, `event_version = 1`.
     - Definir statuses locais: `recorded`, `queued`, `sending`, `sent`, `failed`, `dead_letter`.
     - Incluir campos de relacao operacional: `order_id`, `cart_id`, `payment_attempt_id`, `checkout_completion_log_id`, `payment_intent_id`.
     - Incluir `idempotency_key`, `payload`, `metadata`, `attempt_count`, `last_error_code`, `last_error_message`, `next_retry_at`, `recorded_at`, `queued_at`, `sending_started_at`, `sent_at`, `failed_at`, `dead_lettered_at`.
   - acceptance_criteria:
     - Modelo nao importa services de outros modulos.
     - Modelo nao contem campo raw payload.
     - Modelo nao contem campo de PostHog como requisito de negocio.

2. Definir indexes/constraints e migration draft
   - type: implementation
   - files:
     - `apps/backend/src/modules/analytics-event-log/models/analytics-event-log.ts`
     - `apps/backend/src/modules/analytics-event-log/migrations/Migration*.ts`
   - action:
     - Criar unique em `event_name + idempotency_key`.
     - Criar guard unique/logico por `event_name + order_id`.
     - Criar indexes por `status + next_retry_at`, `order_id`, `payment_attempt_id`, `checkout_completion_log_id` e `payment_intent_id`.
     - Gerar ou escrever migration draft sem aplicar migration real.
   - acceptance_criteria:
     - `medusa db:migrate` nao e executado neste plano.
     - Migration fica como draft revisavel.

3. Criar helpers puros de idempotency e payload
   - type: implementation
   - files:
     - `apps/backend/src/modules/analytics-event-log/service.ts`
     - `apps/backend/src/modules/analytics-event-log/types.ts`
   - action:
     - Implementar `buildPurchaseCompletedIdempotencyKey({ payment_intent_id })`.
     - Retornar `purchase_completed:stripe:{payment_intent_id}`.
     - Implementar builder allowlist-only do payload minimo.
     - Implementar sanitizacao/rejeicao de metadata proibida.
   - acceptance_criteria:
     - Helper rejeita `payment_intent_id` vazio.
     - Payload contem apenas dados permitidos em `07-CONTEXT.md`.
     - Payload nao aceita `client_secret`, Stripe raw payload, Pix QR/copy-paste, PII, Gelato payload ou refund data.

4. Criar testes unitarios do contrato
   - type: test
   - files:
     - `apps/backend/src/modules/analytics-event-log/__tests__/analytics-event-log.unit.spec.ts`
   - action:
     - Cobrir idempotency key, statuses, allowlist e proibicoes de payload.
     - Cobrir shape de indexes/migration em teste fonte quando aplicavel.
   - acceptance_criteria:
     - Testes falham se payload proibido for persistivel.
     - Testes falham se status fora do vocabulario for aceito.

</tasks>
