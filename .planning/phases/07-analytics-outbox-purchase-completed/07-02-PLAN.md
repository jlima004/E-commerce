# 07-02 - Transactional `purchase_completed` Recording And Local Downstream Gate

<objective>
Gravar `purchase_completed` localmente e de forma idempotente no fluxo aceito de criacao de `Order`, preservando a regra de nascimento da `Order` e definindo o gate local para downstream.
</objective>

<scope>
Este plano e para execucao futura. Durante o planejamento atual, nao executar este plano.
</scope>

<tasks>

1. Integrar o outbox ao entrypoint pos-webhook de Order
   - type: implementation
   - files:
     - `apps/backend/src/workflows/order/webhook-order-entrypoint.ts`
     - `apps/backend/src/modules/analytics-event-log/service.ts`
   - action:
     - Resolver o modulo `AnalyticsEventLog` no container.
     - Depois de `Order` confirmada, `CheckoutCompletionLog.completed` e `PaymentAttempt.order_id` correlacionado, criar/reusar `purchase_completed`.
     - Usar `payment_intent_id` como base da idempotency key.
     - Registrar `status = recorded`.
     - Nao chamar PostHog.
   - acceptance_criteria:
     - `Order` continua nascendo somente pelo entrypoint interno pos-webhook.
     - Replay retorna/reusa evento local existente.
     - Falha de PostHog nao existe neste plano porque PostHog ainda nao e chamado.

2. Implementar recovery para Order existente sem outbox
   - type: implementation
   - files:
     - `apps/backend/src/workflows/order/webhook-order-entrypoint.ts`
     - `apps/backend/src/modules/analytics-event-log/service.ts`
   - action:
     - Nos caminhos `reused_existing_order` / recovery da Phase 06, verificar existencia de `purchase_completed`.
     - Se ausente, criar evento local idempotentemente antes de retornar sucesso.
     - Se existente com qualquer status local valido, tratar como gate satisfeito.
   - acceptance_criteria:
     - `Order` existente sem outbox e curada por retry.
     - Duplicidade `event_name + order_id` e tratada como idempotencia ou conflito seguro.

3. Criar helper de gate local para downstream futuro
   - type: implementation
   - files:
     - `apps/backend/src/modules/analytics-event-log/service.ts`
     - `apps/backend/src/modules/analytics-event-log/types.ts`
   - action:
     - Implementar helper planejado como `isPurchaseCompletedLocallyRecorded` ou equivalente.
     - Aceitar statuses duraveis locais: `recorded`, `queued`, `sending`, `sent`, `failed`, `dead_letter`.
     - Rejeitar apenas ausencia de evento local valido.
   - acceptance_criteria:
     - Helper nao consulta PostHog.
     - Helper nao exige `status = sent`.

4. Criar/atualizar testes unitarios e HTTP
   - type: test
   - files:
     - `apps/backend/src/workflows/order/__tests__/webhook-order-analytics-outbox.unit.spec.ts`
     - `apps/backend/integration-tests/http/stripe-webhook-order-creation.spec.ts`
   - action:
     - Cobrir sucesso, replay, concorrencia e recovery.
     - Cobrir ausencia de outbox para estados nao elegiveis.
     - Cobrir que payload nao contem dados proibidos.
   - acceptance_criteria:
     - Uma `Order` confirmada gera exatamente um evento local.
     - Replays/concurrency nao duplicam `Order` nem evento.
     - Downstream gate nao depende de PostHog.

</tasks>
