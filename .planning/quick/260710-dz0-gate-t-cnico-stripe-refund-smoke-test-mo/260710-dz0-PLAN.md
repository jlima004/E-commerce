# Gate técnico — Stripe refund smoke test mode

## Objetivo

Executar, após aprovação do gate ajustado, um refund parcial de 100 centavos
(R$ 1,00) no Stripe test mode e validar a confirmação canônica por webhook sem
cancelamento automático do Order, duplicação de `purchase_completed` ou chamada
real ao Gelato.

## Parada obrigatória do preflight

O preflight encontrou divergência material entre a consulta proposta e o modelo
real. A tabela Medusa `refund` existe, mas não possui `order_id`, `status` ou
`currency_code`; ela contém `amount` e `payment_id`. O contrato financeiro
customizado desta aplicação é `refund_request`, com `order_id`,
`payment_intent_id`, `payment_attempt_id`, `stripe_refund_id`, `amount`,
`currency_code` e `status`.

Não existe `refund_request` para o Order alvo. O entrypoint do webhook procura
primeiro um `refund_request` correlacionável por `stripe_refund_id` ou
`payment_intent_id` + `amount`; sem ele, retorna
`REFUND_WEBHOOK_REQUEST_NOT_FOUND` com classificação `ignored`.

Por isso, o refund não foi criado. Um refund direto no Stripe neste estado não
testaria o fluxo pretendido.

## Gate ajustado para execução posterior

### Tarefa 1 — Criar a intenção local pelo contrato Admin

Antes do Stripe, criar um `refund_request` por
`POST /admin/refunds/request`, nunca por INSERT manual, com:

```json
{
  "order_id": "order_01KX4JDNE9NTXV4YA11Q5MENX9",
  "amount": 100,
  "currency_code": "brl",
  "idempotency_key": "refund-smoke:order_01KX4JDNE9NTXV4YA11Q5MENX9:100:20260710",
  "reason": "requested_by_customer",
  "metadata": {
    "source": "stripe_test_mode_smoke"
  }
}
```

Pré-condições:

- `ADMIN_REFUND_REQUEST_ENABLED=true` no runtime alvo;
- autenticação Admin válida;
- resposta 201 (ou 200 somente em replay da mesma idempotency key);
- `refund_request.status=requested`, `stripe_refund_id=null`, `amount=100`,
  `currency_code=brl`;
- disponibilidade inicial: capturado 9900, confirmado 0, reservado 100,
  disponível 9800.

O endpoint atual apenas cria a reserva local; ele não chama Stripe.

### Tarefa 2 — Criar o refund no Stripe test mode

Somente depois da Tarefa 1, criar pelo conector Stripe:

```text
payment_intent = pi_3TrR4yQy1Qutz95t1baN4nbv
amount = 100
reason = requested_by_customer
```

Confirmar novamente que o objeto/URL pertence a test mode. Registrar o
`stripe_refund_id` retornado e aguardar os webhooks. Não assumir ordenação de
eventos; a documentação Stripe orienta tratar `refund.updated`/`refund.failed`
como atualização do objeto refund e não usar `charge.refunded` como verdade
financeira independente.

### Tarefa 3 — Snapshot posterior e critérios

Critérios canônicos deste runtime:

- Stripe Refund: `status=succeeded`, `currency=brl`, `amount=100`,
  `payment_intent` igual ao PI alvo.
- `refund_request`: `stripe_refund_id` correspondente, `status=confirmed`,
  `amount=100`, `currency_code=brl`, `confirmed_at` preenchido.
- `webhook_event_log`: `refund.created` processado como não-finalizante e
  evento terminal do objeto refund (`refund.updated`, ou `refund.failed` se
  houver falha) processado exatamente uma vez; `charge.refunded`, se recebido,
  é apenas informacional/idempotente.
- Order: metadata `payment_status=partially_refunded`,
  `order_status=confirmed`, `canceled_at=null`.
- `payment_attempt`: continua `payment_confirmed_by_webhook`, com valor e moeda
  originais intactos.
- Analytics: continua existindo exatamente um `purchase_completed`.
- E-mail: não existe fluxo de e-mail de refund no runtime atual. O critério
  executável é nenhum novo `email_delivery_log` para o Order; não alegar
  `skipped/ignored` específico de refund sem implementação correspondente.
- Gelato: continua sem novo `gelato_fulfillment` e sem chamada real.

Observação importante: o handler customizado atualiza `refund_request` e a
metadata financeira do Order; ele não grava a tabela Medusa `refund` nem
recalcula `payment_collection.refunded_amount`. Se “Payment recalculado” exigir
esses campos core, isso é uma lacuna de runtime fora deste smoke e deve virar um
gate de implementação separado antes do refund.

## SQL ajustado

```sql
select
  id, order_id, payment_intent_id, payment_attempt_id, stripe_refund_id,
  idempotency_key, amount, currency_code, status,
  confirmed_at, failed_at, canceled_at, rejected_at, created_at, updated_at
from refund_request
where order_id = 'order_01KX4JDNE9NTXV4YA11Q5MENX9'
order by created_at desc;
```

```sql
select
  pa.id as payment_attempt_id,
  pa.payment_collection_id,
  opc.order_id,
  pc.status as payment_collection_status,
  pc.captured_amount,
  pc.refunded_amount,
  pc.currency_code,
  p.id as payment_id,
  p.amount as payment_amount,
  p.currency_code as payment_currency_code,
  p.provider_id,
  p.captured_at,
  p.canceled_at,
  p.data->>'id' as provider_payment_id,
  r.id as refund_id,
  r.amount as refund_amount,
  r.payment_id as refund_payment_id,
  r.created_at as refund_created_at,
  r.updated_at as refund_updated_at
from payment_attempt pa
left join order_payment_collection opc
  on opc.payment_collection_id = pa.payment_collection_id
 and opc.deleted_at is null
left join payment_collection pc
  on pc.id = pa.payment_collection_id
 and pc.deleted_at is null
left join payment p
  on p.payment_collection_id = pa.payment_collection_id
 and p.deleted_at is null
left join refund r
  on r.payment_id = p.id
 and r.deleted_at is null
where pa.id = 'payatt_de07f78b4e094f77'
order by r.created_at desc nulls last;
```
